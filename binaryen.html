Compiling to
WebAssembly
with Binaryen


Alon Zakai / @kripken

June 2016

WebAssembly (wasm) is a new binary executable format


Will allow even large compiled codebases to run efficiently on the web


Main initial focus is on C and C++, since those are very popular (e.g., in the games space: Unity, Unreal, etc.)

Using emscripten and LLVM+clang, currently two options:


C++  ⇒  clang  ⇒  asm.js backend  ⇒  asm2wasm  ⇒  wasm


C++  ⇒  clang  ⇒  wasm backend  ⇒  s2wasm  ⇒  wasm


asm2wasm and s2wasm are part of the Binaryen project

Binaryen handles wasm-specific tasks like binary generation and "compression" operations (that might not make sense to do in LLVM)

Why two options?


asm.js backend already existed, asm2wasm easy to add


This option already works and is pretty stable

The wasm backend is still in development, but aims to avoid some asm.js limitations:


Full i64 support


Supports all LLVM IR (full legalization)


LLVM's codegen can emit very good code


Developed upstream

However, wasm backend compiles more slowly, due to known LLVM codegen issues (e.g., WebKit's recent switch away from LLVM)


For example, [NUMBERS]


Incremental compilation will help both backends, but proportionally, so difference will remain


⇒ power of LLVM codegen comes at a cost

For C++, you may be using clang+LLVM anyhow


But for other languages, things are more open, with multiple options and tradeoffs possible


One option is to compile directly to Binaryen:

your compiler  ⇒  binaryen  ⇒  wasm

your compiler  ⇒  binaryen  ⇒  wasm


Let's see some of the benefits of doing this

Binaryen can handle both wasm-specific tasks (emit the binary format, compression, etc.) and general optimization


So you might want it anyhow for the former

Binaryen has minimal, compact data structures


E.g., no parent/context pointers


Cache-friendly, fast to traverse


In some cases things need to be recomputed, but tradeoff often worth it


Similar philosophy to the B3 JIT which replaced LLVM in WebKit

Binaryen's IR is simpler than LLVM's (e.g., no arbitrary-sized ints)


Limits some optimizations, but passes run faster

In fact Binaryen's IR is basically WebAssembly itself


Optimize and emit wasm using a single IR: avoid the overhead of constructing and translating between multiple IRs


(may need to change this eventually; let's see how far we can get without it)

As a result of the previous points, Binaryen is much smaller than LLVM

LLVM is single-threaded


Binaryen can both build IR, and optimize it, using multiple cores


Can also pipeline: optimize ready functions while others are still being generated


Binaryen functions do not refer to global state, making this straightforward

Some of LLVM's power is irrelevant anyhow to a portable and relatively high-level target like Binaryen


No need for register allocation


LLVM is great at optimizing the last bit of speed from x86 specifics, etc., but we don't need any of that either

compile times, code size, benchmarks

Binaryen is also very simple to use


Input can be structured or as a general control flow graph


Input can be in SSA or register form


In both cases the Binaryen optimizer can transform the code into something good

There is a C API, can also use the C++ API if you want


// C API example
BinaryenModuleRef module = BinaryenModuleCreate();
// Creation a function type for  i32 (i32, i32)
BinaryenType params[2] = { BinaryenInt32(), BinaryenInt32() };
BinaryenFunctionTypeRef iii =
  BinaryenAddFunctionType(module, "iii",
                          BinaryenInt32(), params, 2);
// Create a function that adds two numbers
BinaryenExpressionRef
  x = BinaryenGetLocal(module, 0, BinaryenInt32()),
  y = BinaryenGetLocal(module, 1, BinaryenInt32());
BinaryenExpressionRef add =
  BinaryenBinary(module, BinaryenAddInt32(), x, y);
BinaryenFunctionRef adder =
  BinaryenAddFunction(module, "adder", iii, NULL, 0, add);
BinaryenModuleOptimize(module);
// Can use BinaryenModuleWrite() to emit binary etc.

Proof of concept of Rust MIR ⇒ Binaryen: mir2wasm


rust  ⇒  rust optimizer  ⇒  binaryen optimizer  ⇒  wasm


Will be able to benefit from both language-specific optimizations (which more languages are adding these days, e.g., Swift), as well as generic optimizations


Should have very good codegen times, but still early. Help is welcome!
