Compiling to
WebAssembly
with Binaryen


Alon Zakai / @kripken

June 2016

Background


WebAssembly (wasm) is a new binary executable format


Will allow even large compiled codebases to run efficiently on the web, even better than asm.js


Main initial focus is on C and C++, since those are very popular (e.g., in the games space: Unity, Unreal, etc.)

Using emscripten and LLVM+clang, currently two options:


C++  ⇒  clang  ⇒  asm.js backend  ⇒  asm2wasm  ⇒  wasm


C++  ⇒  clang  ⇒  wasm backend  ⇒  s2wasm  ⇒  wasm


asm2wasm and s2wasm are part of the Binaryen project

Binaryen


A compiler and toolchain library for WebAssembly, in C++


Focuses on wasm-specific tasks like reading and writing the binary format, as well as compression to minimize downloads


Also has general-purpose optimizations like coloring, dead code elimination, etc.

Binaryen provides the asm2wasm and s2wasm tools mentioned before, for the asm.js and wasm backends respectively


Why two backends?


asm.js backend already existed, asm2wasm easy to add


asm2wasm already works and is pretty stable

The wasm backend is still in development, but should be better than the asm.js one:


Full i64 support


Supports all LLVM IR (full legalization)


LLVM's codegen can emit very good code


Developed upstream

However, wasm backend compiles more slowly, due to known LLVM codegen issues (e.g., WebKit's recent switch away from LLVM)


Incremental compilation will eventually help make compile time matter less


But it still does, in particular LTO-type builds which matter a lot on the web


Power of LLVM codegen comes at a cost

For C++, you may be using clang+LLVM anyhow, so wasm backend is a natural fit


But if we focus now on other languages, your options may be more open

One option is to use Binaryen as your compiler backend, without LLVM or something else:

your compiler  ⇒  binaryen  ⇒  wasm


Let's see some of the benefits of doing this

Binaryen can handle wasm-specific tasks (emit the binary format, compression, etc.) and general optimization


So you might want it anyhow for the wasm-specific tasks


Then your compiler can avoid an extra dependency on something else for general optimizations

Binaryen is much smaller than LLVM


If you need all of LLVM's power, definitely worth it, but you don't always

Speed: Binaryen has minimal, compact data structures


E.g., no parent/context pointers


Cache-friendly, fast to traverse


Less general optimization framework than LLVM, and in some cases things need to be recomputed (use lists)


But tradeoff often worth it (similar philosophy to the B3 JIT)

LLVM is single-threaded


Binaryen can both build IR, and optimize it, using multiple cores


Can also pipeline: optimize ready functions while others are still being generated


Binaryen functions do not refer to global state, making this straightforward

Binaryen only compiles to WebAssembly, and WebAssembly is very simple compared to x86, ARM, etc.


WebAssembly is like an abstracted RISC-like arch with structured control flow ⇒ instruction selection is simple


No need for register allocation, scheduling, etc. - WebAssembly VMs do that!


As a result, some of LLVM's power is irrelevant anyhow

Binaryen Optimizer Numbers

Let's look at BananaBread converted to WebAssembly, using the asm2wasm option and Binaryen

Binaryen on unoptimized asm.js:


binaryen optsnoyes
binary size (MB)3.802.46-35%
time (seconds)2.983.60

(less is better on all)


Binaryen's general and wasm-specific optimizations can optimize the code quite a lot


Note wall time increases by just 21%: we optimize in parallel

Binaryen on optimized asm.js (using emscripten's asm.js optimizer):


binaryen optsnoyes
binary size (MB)2.442.16-13%
time (seconds)1.632.13

Less of a benefit since code is already fairly optimized, but still significant

Can also run Binaryen on wasm from the wasm backend:


binaryen optsnoyes
binary size (MB)2.102.05-2.5%

LLVM has excellent general optimizations, as well as wasm-specific ones like coloring and stackification, but a small improvement is still possible

An important takeaway from these numbers is that to emit efficient wasm you need to optimize it


Just like you minify JS or CSS, basically

Partial list of Binaryen's optimization passes

  • Local coloring
  • Local stacking
  • Block/if return value opts
  • Dead code elimination
  • Duplicate function elimination
  • Local reordering
  • ...

With Binaryen you can get all that, and with little extra compile time

Using Binaryen


Binaryen aims to be simple to use


Straightforward C API in one header; can also use C++


Input can be structured (loops, ifs), also an API for general control flow graphs (which we "reloop")


Input in register form; ok if there are tons of locals (e.g., from SSA), we optimize that down

Proof of concept of Rust MIR ⇒ Binaryen: mir2wasm


rust  ⇒  rust optimizer  ⇒  binaryen optimizer  ⇒  wasm


Should have very good codegen times, but still early. Help is welcome!

Summary


Binaryen aims to be a flexible, fast, and easy to use way for your compiler to target WebAssembly


Feedback on using Binaryen with other languages is very welcome, we hope you find it useful!
