Compiling to
WebAssembly
with Binaryen


Alon Zakai / @kripken

June 2016

Background


WebAssembly (wasm) is a new binary executable format


Will allow even large compiled codebases to run efficiently on the web, even better than asm.js


Main initial focus is on C and C++, since those are very popular (e.g., in the games space: Unity, Unreal, etc.)

Using emscripten and LLVM+clang, currently two options:


C++  ⇒  clang  ⇒  asm.js backend  ⇒  asm2wasm  ⇒  wasm


C++  ⇒  clang  ⇒  wasm backend  ⇒  s2wasm  ⇒  wasm


asm2wasm and s2wasm are part of the Binaryen project

Binaryen handles wasm-specific tasks like binary generation and "compression" operations (that might not make sense to do in LLVM)

Why two options?


asm.js backend already existed, asm2wasm easy to add


This option already works and is pretty stable

The wasm backend is still in development, but aims to avoid some asm.js limitations:


Full i64 support


Supports all LLVM IR (full legalization)


LLVM's codegen can emit very good code


Developed upstream

However, wasm backend compiles more slowly, due to known LLVM codegen issues (e.g., WebKit's recent switch away from LLVM)


Incremental compilation will help both backends, but proportionally, so difference will remain


⇒ power of LLVM codegen comes at a cost

For C++, you may be using clang+LLVM anyhow


But for other languages, you have several options


One option is to compile directly to Binaryen:

your compiler  ⇒  binaryen  ⇒  wasm


Basically, this means using Binaryen as a compiler backend


Let's see some of the benefits of doing this

Binaryen can handle wasm-specific tasks (emit the binary format, compression, etc.) and general optimization


So you might want it anyhow for the former, and can avoid an extra dependency

Binaryen has minimal, compact data structures


E.g., no parent/context pointers


Cache-friendly, fast to traverse


In some cases things need to be recomputed, but tradeoff often worth it (similar philosophy to the B3 JIT)

Binaryen's IR is simpler than LLVM's


E.g. no arbitrary-sized ints


Limits some optimizations, but passes run faster

In fact Binaryen's IR is basically WebAssembly itself


Optimize and emit wasm using a single IR: avoid the overhead of constructing and translating between multiple IRs


When this is good enough, it can be very fast


May need to change this eventually; let's see how far we can get without it

Binaryen is much smaller than LLVM


If you need all of LLVM's power, definitely worth it, but you don't always

LLVM is single-threaded


Binaryen can both build IR, and optimize it, using multiple cores


Can also pipeline: optimize ready functions while others are still being generated


Binaryen functions do not refer to global state, making this straightforward

Some of LLVM's power is irrelevant anyhow to a portable and relatively high-level target like WebAssembly


No need for register allocation


LLVM is great at optimizing the last bit of speed from x86 specifics, etc., but wasm doesn't need any of that either

Numbers

Converting BananaBread from asm.js to wasm using Binaryen:

no optswith opts
binary size (MB)3.802.46-35%
time (seconds)2.983.60

(less is better on all)


Time increases by just 21%, since we optimize in parallel


The code optimized here was fairly unoptimized, just raw asm.js backend output

Also interesting to look at more optimized code, asm.js that was optimized by the asm.js optimizer:

no optswith opts
binary size (MB)2.442.16-13%
time (seconds)1.632.13

Less of a benefit since code is already fairly optimized, but still significant

And what about wasm from the wasm backend?

no optswith opts
binary size (MB)2.102.05-2.5%

LLVM's general optimizations don't leave room for much more improvement, but a few percent are still possible

Binaryen is also simple to use


Straightforward C API in one header; can also use the internal C++ API


Input can be structured (loops, ifs) or a general control flow graph (which we restructure and optimize)


Input can be in register form or SSA form (which we color and optimize)

Proof of concept of Rust MIR ⇒ Binaryen: mir2wasm


rust  ⇒  rust optimizer  ⇒  binaryen optimizer  ⇒  wasm


Should have very good codegen times, but still early. Help is welcome!

Summary


Binaryen aims to be a flexible, fast, and easy to use way to compile to WebAssembly


Feedback on using Binaryen with other languages is very welcome, we hope you find it useful!
