Hybrid libc, musl + plus parts written in JavaScript

Emscripten details


We have some JavaScript-specific optimizations that we run on a JavaScript AST

Robustness is comparable to native builds with clang+LLVM: we can run the csmith fuzzer without finding new Emscripten-specific issues

You might be curious at this point what the emitted code looks like...


  // C++
  int func(int *p) {
    int r = *p;
    return calc(r, r << 16);
  }

⇒ Emscripten ⇒


  // JS
  function func(p) {
    var r = HEAP[p];
    return calc(r, r << 16);
  }

Almost direct mapping in many cases

Another example:

  float array[5000]; // C++
  int main() {
    for (int i = 0; i < 5000; ++i) {
      array[i] += 1.0f;
    }
  }

⇒ Emscripten ⇒

  var heap = new ArrayBuffer(32768); // JS
  var g = new Float32Array(heap);
  function main() {
    var a = 0, b = 0;
    do {
      a = 8 + (b << 2) | 0;
      g[a >> 2] = +g[a >> 2] + 1.0;
      b = b + 1 | 0;
    } while ((b | 0) < 5000);
  }

This "style" of code is a subset of JS called asm.js, which we'll discuss more later

So that's what the code can look like. But there are some fundamental differences here...

Build Portability


C++
Need to recompile for another CPU or OS
JS
Single build runs the same everywhere


Strict portability prevents some optimizations

Sandboxing


C++
Applications can use the system libs, access the local filesystem, etc.
JS
Sandboxed, cannot see the machine it is running on


Applications must ship their own system libraries

We "fake" a filesystem to make porting easy

Numeric Types


C++
char, short, int, int64, float, double
JS
double


We build for a 32-bit target, because 64-bit integers cannot all fit in doubles (but 32-bit ones can)

Performance Model


C++
C-style code maps closely to CPU, higher-level C++ aspects can use RAII, etc., giving predictability
JS
virtual machine (VM), just in time (JIT) compilers w/ type profiling, garbage collection, etc.

Yet, we need to be able to run code at reasonable speeds in a predictable way, or else this is pointless...

JS began as a slow interpreted language


Competition ⇒ type-specializing JITs


Those are very good at implicitly statically typed code


  var x = 10;
  var y = 20;
  var z = (x + y) | 0; // int32 addition!

That's what asm.js is: a subset of JavaScript where all the operations are easily seen to actually be statically typed

  var heap = new ArrayBuffer(32768);
  var heap8 = new Int8Array(heap);
  var heap16 = new Int16Array(heap);
  var heap32 = new Int32Array(heap);

  function mem_access() {
    return heap32[heap8[100] >> 2];
  }

Loads in C++ become loads in LLVM IR which become reads from typed arrays in JS, which become reads in machine code


Emscripten's memory model is identical to LLVM's, including aliasing, so can use all LLVM opts

Emscripten supports exceptions... differently:

  // C++
  void func() {
    try {
      something();
    } catch (Type T) {
      handle(T);
    }
  }

⇒ Emscripten ⇒

  // JS
  void func() {
    invoke(10); // call a function pointer, checking for throw
    var T = get_thrown();
    if (T) {
      if (can_handle(T, 400)) { // 400 -> typeid of Type
        handle(T);
      } else {
        do_throw(T);
      }
    }
  }

...what do those utility function do? 
  // JS
  function invoke(ptr) {
    try {
      __thrown__ = 0;
      dyn_call(ptr);
    } catch (e) {
      __thrown__ = e;
    }
  }
  function do_throw(ptr) {
    throw T;
  }
  function can_handle(ptr, type) {
    // call into libcxxabi internals
  }

We implement C++ exceptions using JavaScript exceptions

JavaScript provides call stack unwinding, we just hook into libc++abi code to check what to catch

Performance of C++ exceptions using Emscripten therefore depends on the performance of JavaScript exceptions

Back to Memory


As mentioned, we represent memory using a single flat array

Pointers are indexes into the array


  var HEAP = new Uint8Array(20*1024*1024);
  function compiledCode(ptr) {
    HEAP[ptr] = 12; // write to an address
    return HEAP[ptr + 4]; // read from an address
  }

Which is basically how C and C++ see memory - a pointer can point anywhere in all of memory

But this is not how languages like JavaScript, C#, Java, Python etc. see memory

Each object or array in those languages is in its own "space", which is even bounds-checked, and there is no pointer that can point to anywhere, pointers are references to objects

This isn't just an academic point!


  // same as before
  var HEAP = new Uint8Array(20*1024*1024);
  function compiledCode(ptr) {
    HEAP[ptr] = 12;
    return HEAP[ptr + 4];
  }
  function getInput(array) {
    // array is a NORMAL JavaScript array
    // compiledCode cannot refer to it!
    // must copy into the HEAP
    var buffer = malloc(array.length);
    HEAP.set(array, buffer);
    // buffer is now a pointer to a copy of array
    return compiledCode(buffer);
  }

These problems happen when connecting compiled code and outside JavaScript
