Emscripten's approach


Emit low-level JavaScript, close to LLVM IR

No JS objects, arrays, closures, etc.

Yes normal JS calls, vars, etc.


  // LLVM IR
  define i32 @func(i32* %p) {
    %r = load i32* %p
    %s = shl i32 %r 16
    %t = call i32 @calc(i32 %r, i32 %s)
    ret i32 %t
  }

⇒ Emscripten ⇒


  // JS
  function func(p) {
    var r = HEAP[p];
    return calc(r, r << 16);
  }

Another example:

  float array[5000]; // C++
  int main() {
    for (int i = 0; i < 5000; ++i) {
      array[i] += 1.0f;
    }
  }

⇒ Emscripten ⇒

  var g = Float32Array(5000); // JS
  function main() {
    var a = 0, b = 0;
    do {
      a = b << 2;
      g[a >> 2] = +g[a >> 2] + 1.0;
      b = b + 1 | 0;
    } while ((b | 0) < 5000);
  }

(this "style" of code is a subset of JS called asm.js)

JS as a compilation target


JS began as a slow interpreted language

Gained type-specializing JITs

Those are very good at statically typed code

LLVM compiled through Emscripten is exactly that

Performance

(VMs and Emscripten from July 10th 2013, run on 64-bit linux)

Why this is fast


(x+1)|0   ⇒   32-bit integer + in modern JS VMs

Loads in LLVM IR become reads from typed array in JS, which become reads in machine code

Emscripten's memory model is identical to LLVM's (flat C-like, aliasing, etc.), so can use all LLVM opts


Open source (MIT/LLVM)

Began in 2010

Written mostly in JS, some Python, some C++

Bulk of codebase is libraries + toolchain + test suite, core compiler is a small fraction of those

Wait, that's not an LLVM backend..?

3 JS compilers, 3 ways


Mandreel: Typical LLVM backend, uses tblgen, selection DAG (like x86, ARM backends)

Duetto: Processes LLVM IR in llvm::Module (like C++ backend)

Emscripten: Processes LLVM IR in assembly

Emscripten's choice


JS is such an odd target ⇒ wanted maximal flexibility in codegen


Helped prototype & test many approaches

non-backend downsides


Emscripten currently must do its own legalization


Emitted code is very low-level - perhaps close to what an LLVM backend could emit?

Optimizing JS


Emscripten has 3 crucial optimizations for JS, that might be irrelevant for other LLVM backends:

Expressionize, Registerize and Reloop

1. Expressionize


  var a = g(x);
  var b = a + y;
  var c = HEAP[b];
  var d = HEAP[20];
  var e = x + y + z;
  var f = h(d, e);
  FUNCTION_TABLE[c](f);

⇒


  FUNCTION_TABLE[HEAP[g(x) + y](h(HEAP[20], x + y + z));

1. Expressionize


Reduces code size

Improves JIT time and execution speed: fewer variables ⇒ less stuff for JS engines to worry about

2. Registerize


  var a = g(x) | 0; // integers 
  var b = a + y | 0;
  var c = HEAP[b] | 0;
  var d = +HEAP[20]; // double

⇒


  var a = g(x) | 0;
  a = a + y | 0;
  a = HEAP[a] | 0;
  var b = +HEAP[20];

2. Registerize


Looks like regalloc, but goal is different: Minimize # of total variables (in each type), not spills

JS VMs will do regalloc, only they know the actual # of registers

Benefits code size & speed like expressionize

3. Reloop

  block1:
    ; code1
    br i1 %cond, %label block1, label %block2

  block2:
    ; code2
    br %label block1

Without relooping (emulated gotos):

  var label = 0;
  while (1) switch (label) {
    case 0:
      // code1
      label = cond ? 0 : 1; break;
    case 1:
      // code2
      label = 0; break;
  }

3. Reloop

  block1:
    ; code1
    br i1 %cond, %label block1, label %block2

  block2:
    ; code2
    br %label block1

With relooping:


  while (1) {
    do {
      // code1
    } while (cond);
    // code2
  }

3. Reloop


Relooping allows JS VM to optimize better, as it can understand control flow

Also reduces code size

Emscripten Relooper code is generic, written in C++, and used by other LLVM-based projects like Duetto

Opts Summary


Each of expressionize, registerize and reloop can make a big impact on perf - anywhere from 1x-10x faster

2 of the 3 require a precise modelling of JS semantics (and order of operations is in some cases surprising!)

Is there a nice way to do that in an LLVM backend, or do we need a JS AST?

JS is improving


First Emscripten and Mandreel figured out stuff that worked well in existing JS VMs

Later JS VMs added new optimizations for that type of code

Collaboration with JS standards bodies also led to some new language features

New JS features

Math.imul - precise 32-bit multiply (the only integer op without a trivial impl)

Math.fround - round a 64-bit double to a 32-bit float, allows efficient implementation of floats

SIMD - There is a spec, and Emscripten supports LLVM vector types, so both C SIMD intrinsics and LLVM autovectorization will eventually work

Conclusion


LLVM bitcode can be compiled to JavaScript

LLVM bitcode can run in all browers, at high speed, in a standards-compliant way
