The Binaryen Optimizer Goes To 4

Alon Zakai / June 2018

C/C++ compilers - gcc, clang, emcc - have flags like these:


  • -O1: "Optimize"
  • -O2: "Optimize even more"
  • -O3: "Optimize yet more"


(actual quotes from docs)

When you run emcc -O3 main.cpp -o main.html Emscripten does this:


LLVM IR → LLVM -O3 → wasm → Binaryen -O3 → final wasm

Binaryen has been mostly tuned for this LLVM case, and shrinks the final wasm by 15%

But non-LLVM compilers to wasm are important too! Go, AssemblyScript, etc.


Wasm GC will let compile-to-js languages (which mostly do not use LLVM) become compile-to-wasm languages:

the same
source code		 ---→

---→		 JavaScript

WebAssembly

Possibilities here include: TypeScript, Elm, Reason, etc.

To accelerate that, Binaryen helps write compilers to wasm:


  • Simple IR (optional basic block relooping, etc.)
  • Handles wasm binary format details
  • Easy to use:
    • JS bindings (binaryen.js), used by AssemblyScript
    • C bindings, used by Asterius (Haskell)
  • Optimizes & minifies

But some code is harder to optimzie:

(call $report
 (loop $loop (result i32) ;; loop nested in call
  (br_if $loop
   (call $do-work)
  )
  (call $get-result) ;; value flows out through the loop
 )
)

LLVM wouldn't emit stuff like this, but we do want to optimize it well

To help here Binaryen has a "flatten" pass:

(call $report
 (loop $loop (result i32)
  (br_if $loop
   (call $do-work)
  )
  (call $get-result)
 )
)
flatten
---→ 
(loop $loop
 (set_local $0
  (call $do-work)
 )
 (br_if $loop
  (get_local $0)
 )
 (set_local $1
  (call $get-result)
 )
)
(call $report
 (get_local $1)
)

Flat IR is simpler, for example, no nested side effects:

(call $report
 (loop $loop (result i32)
  (br_if $loop
   (call $do-work)
  )
  (call $get-result)
 )
)
flatten
---→ 
(loop $loop
 (block
  (set_local $0
   (call $do-work)
  )
  (br_if $loop
   (get_local $0)
  )
 )
 (set_local $1
  (call $get-result)
 )
)
(call $report
 (get_local $1)
)

Binaryen Optimizer Design

  • Binaryen IR is almost identical to wasm, to allow efficient minification
  • No separate "flat" and "full" IRs; flat IR is just a subset
  • Most passes work on all IR (but may be more effective if flat)
  • A few passes depend on flat IR (when hard to write a general pass)

So most passes are written normally, and if you flatten before running them you can get a boost

That boost takes Binaryen up to 4:


-O4 = flatten + flat-only passes + -O3

Fully optimizes a bunch of testcases we noticed in AssemblyScript

Not much large non-LLVM code yet to test on, but on fuzz testcases, -O4 shrinks an extra 20%

But also 3x slower compilation - shows importance of option to run without flattening!

So far Binaryen goes up to 4

Maybe someday we'll go up to 11
