You are running a cross-compiler: Generating code for another platform than the one you are currently on

Similar to developing a mobile game on a desktop!

Cross-compiling

Cannot use system headers or libraries

emcc uses the emscripten headers and libs, which are portable

Need a library? Need to build it to bitcode using emcc (or get that from someone that already did it), then statically link with your project

Undefined Behavior

A common issue with cross-compiling, for example:

  char buffer[8];
  int *i = (int*)&buffer[1]; /* unaligned! */
  *i = 10;

This works on x86, can fail on ARM

JavaScript, like ARM, will not produce correct results

Build with emcc SAFE_HEAP to get runtime checks, makes debugging this easy

Undefined Behavior

Event Loop

Code on the web must run as a sequence of short-running events

Anything long-running will pause the browser, lead to dreaded "slow script dialogue" :(

Dialouge boxes look nicer today, but problem remains the same

Bad :(


  int main() {
    init();
    while (is_game_running()) {
      do_frame();
    }
    return 0;
  }

Good :)


  void do_web_frame() {
    if (!is_game_running()) {
      emscripten_cancel_main_loop();
      return;
    }
    do_frame();
  }

  int main() {
    init();
    emscripten_set_main_loop(do_web_frame, 0, 0);
    return 0;
  }

See <emscripten.h> for more details

Getting a Window, I/O, etc.

Emscripten supports a few C APIs for this:

HTML5, SDL, glut, glfw

Simple stuff works in all, complex stuff varies (list starts from best)

Can use SDL, glut etc. for existing codebases

HTML5 is a good option for new codebases or if you know HTML5 well

html5.h


  #include <emscripten.h>
  #include <emscripten/html5.h>

  int main() {
    emscripten_set_canvas_size(1024, 768);
    emscripten_set_keydown_callback(0, 0, 1, key_callback);
    emscripten_set_main_loop(do_web_frame, 0, 0);
    return 0;
  }

SDL


  #include <SDL.h>
  #include <emscripten.h>

  int main() {
    SDL_Init(SDL_INIT_VIDEO);
    SDL_Surface *screen =
      SDL_SetVideoMode(1024, 768, 32, SDL_HWSURFACE);
    emscripten_set_main_loop(do_web_frame, 0, 0);
    /* use SDL_PollEvent to receive input etc. */
    return 0;
  }

Debugging

SDL, glut etc. are good to use because you can build the same code both to native and web

One approach is to then use your normal debugging tools on the native build

Debugging

Also possible to debug web builds

emcc has various levels of debuggability of generated code

Debugging

Default optimized (-O1 and above) output is minified

Try to debug this...


  function a(a,b){a=a|0;b=b|0;f(a+b|0);}

...you're gonna have a bad time

Debugging

-g1 : preserve whitespace


  function a(a, b) {
    a = a | 0;
    b = b | 0;
    f(a + b | 0);
  }

Debugging

-g2 : preserve function names


  function _addAndPrint(a, b) {
    a = a | 0;
    b = b | 0;
    _printAnInteger(a + b | 0);
  }

Debugging

-g3 : preserve variable names


  function _addAndPrint($left, $right) {
    $left = $left | 0;
    $right = $right | 0;
    _printAnInteger($left + $right | 0);
  }

They will not match original source variable names 1-1, but are often quite close

Rendering

Emscripten supports OpenGL

The subset of OpenGL ES 2.0 that is parallel to WebGL (ES 2.0 minus clientside data & a few other things) is best: fast, fully supported

General OpenGL ES 2.0 works well, using the FULL_ES2 flag (emulating clientside data adds some overhead)

Some OpenGL 1.x stuff works using the LEGACY_GL_EMULATION flag (but much doesn't, and overhead is significant)

WebGL via GL ES 2.0


void do_web_frame() {
  /* normal GLES 2.0 code, no clientside data */
  glClear(GL_DEPTH_BUFFER_BIT);

  glVertexAttribPointer(1, 3, GL_FLOAT, GL_FALSE, 24, 0);
