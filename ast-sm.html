wasm moving to require explicit drop


Spec can be written equivalently as either AST + explicit drop or stack machine + restrictions

  // binary format
  block-begin // block markers before and after,
  i32.const 1 // but most stuff is basically a stack machine
  get_local 3
  i32.add
  drop        // value is not used, so drop it from the stack
  block-end
  // AST representation
  (block
    (drop ; AST could in principle work without this
      (i32.add
        (i32.const 1)
        (get_local 3)
      )
    )
  )
  // natural text format
  {
    1 + $3; // maybe add drop()?
  }

Suggestions have come up to eliminate these restrictions (now or later)


That would mean wasm is a more general stack machine, and no longer equivalent to an AST


We would lose some of the current structure guarantees in return for flexibility

  block-begin
  ..
  i32.const 0
  i32.const 1
  i32.store

In a restricted stack machine, we know at this point that the store is on the current block, and that nothing after it can pop stuff before it


I.e., the AST structure means non-returning operators like store, set_local, etc. are "midpoint markers" in the block, they separate the block into independent units

Discussing with Lars, midpoint marker optimization could help baseline JITs


Regalloc etc. context tracked til the end of the block could be flushed earlier (better for locality, etc.)

Opcodes after a midpoint are very limited: nothing can pop, so no binaries or unaries, etc. Probably most are get_local or const.


Can experiment with replacing this flow:

  }

with

  if (nextOpcode == get_local) ..
  else ..the switch..

"Layer 1" compression would start as userspace, and hopefully get standardized later


Limited opcodes after midpoint markers can help compression


Imagine compression that uses optimal bits at each point, like JSZap, then after each midpoint marker we need far fewer bits
